# Skill: Open Source Project Analysis

## 适用场景

用户提供 GitHub 仓库、截图、文章、海报或社媒传播内容，希望理解一个开源项目为什么火、技术上怎么做、解决什么问题、能达到什么效果，以及是否值得学习、试用、复刻或改造成自己的工作流。

## 分析流程

1. **确认输入和目标**
   - 记录仓库 URL、截图/海报、用户关注点和目标场景。
   - 如果用户没有说明目标，默认从“技术学习、内容传播、产品机会、工作流复刻”四个角度分析。

2. **核对 GitHub 事实**
   - 核对 stars、forks、release、license、主要语言、README 功能描述、安装方式、配置项和维护状态。
   - 将仓库事实与截图/文章中的数字和说法对比，标记过期信息或夸张表达。

3. **拆解传播叙事**
   - 提炼标题钩子、核心卖点、目标受众、情绪触发点和行动号召。
   - 区分事实描述、价值判断、营销话术和未验证承诺。

4. **拆解技术工作流**
   - 按输入、处理模块、外部依赖、状态流转、输出物拆解项目。
   - 标记哪些模块适合自动化，哪些环节需要人工审核。
   - 关注接口、配置、失败重试、成本、延迟、隐私和版权风险。

5. **判断解决的问题**
   - 从用户原有流程出发，说明该项目减少了哪些人工步骤、降低了哪些门槛、提高了哪些效率。
   - 避免将“生成结果”直接等同于“商业结果”。

6. **评估可达到的效果**
   - 分别说明对创作者、开发者、运营者、团队或创业者的实际用途。
   - 标明效果成立所需前提，例如数据回收、人工质检、平台适配、素材授权和持续迭代。

7. **给出行动建议**
   - 输出试跑路径、最小实验、采用条件、放弃条件和下一步验证。
   - 如果适合做 Agent，给出 Agent 角色、输入、步骤、输出和协作员工。

## 证据等级

- **已验证事实**：来自仓库、release、代码、文档、截图或实际命令输出。
- **合理推断**：基于项目结构、文档说明或常见工程模式推导，但未直接验证。
- **营销表达**：用于传播或吸引注意力，不能直接当作事实。
- **未验证假设**：缺少证据，需要试跑、测试或用户补充材料。

## 标准输出

- 背景理解
- GitHub 事实核对
- 图片/传播叙事拆解
- 核心工作流
- 解决的问题
- 可达到的效果
- 自动化与人工审核边界
- 采用/学习/复刻建议
- 风险/疑问
- 下一步行动
